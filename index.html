Virga

http://docs.virga.service.vpc-us-east-1.github.net/

Virga is the infrastructure powering Machine Learning projects at GitHub. At its core, Virga is an implementation of Argo with related services that allow users to leverage resources on our kubernetes cluster.

Context

A machine learning project can touch a wide swath of areas and require a large number of skillsets: from data collection and logging, data munging and feature extraction, model building and hyper-parameter tuning, to production-grade API development for serving. A more complete list of components necessary for an ML project can be seen below.

The sheer number of tasks necessary for a full pipeline can be overwhelming for a single ML engineer or data scientist who just wants to build models to serve a product need. Virga aims to decrease friction and lower the time to production for ML applications by automating or abstracting away as many of these tasks as possible, as well as provide a flexible, easy-to-use, and powerful platform for components which cannot be abstracted.

Virga Supporters and Maintainers

Virga is maintained by members from the @github/data-science-foundations team and they are responsible for the core tools and infrastructure, but many ancillary tools and documentation come from Virga’s users. In many ways, all of the data org is a maintainer of Virga.

You can see the current development status of the platform by following the project board, or following open Issues in this repo.

Platform

The Virga platform has a number of different components, all working together to enable different parts of the ML development workflow. The following sections will give a sense of the different capabilities the platform provides, and can guide you towards individual tools that might be able to help your specific ML project.

For information on Service Expectations, checkout the Service Level Agreement Doc.

Kubernetes

All of Virga infrastructure is built on top of Kubernetes. This powerful container-based platform is capable of deploying any number of custom services to be used in service of ML projects. While Kubernetes has quality documentation, Virga houses more complete walkthroughs for how to deploy specific types of services on the platform.

Virga components can be managed by the end user in the UI.

Argo

IF YOU HAVE TROUBLE ACCESSING ARGO - RUN scripts/argo-bootstrap AS A FIRST ATTEMPT TO TROUBLESHOOT.

One of the core components of Virga is Argo, which is a flexible workflow engine built on top of kubernetes. Think something like airflow, but can also handle compute, making it perfect for ML tasks like feature processing or model training. No longer do ML engineers or data scientists have to rely on adhoc exploration hosts to run scheduled re-trainings, or have to resort to Spark to do intensive compute operations. Argo can vastly reduce the time it takes to move exploratory scripts into automated pipelines for ML applications.

Argo can also be used in conjunction with Airflow via a custom ArgoOperator. This will allow you to add the compute power of Virga’s infrastructure alongside any warehouse queries you’re used to running in Airflow.

To get started with Argo, read through the documentation in the argo/ directory.

Why “Virga” the name?

In meteorology, virga is an observable streak or shaft of precipitation falling from a cloud that evaporates or sublimates before reaching the ground. @inc0 thought this was an apt metaphor for this new platform: a place where all ML work can be solely contained within a cloud infrastructure.

It was a good enough name, and there were no other challengers, so it stuck.
